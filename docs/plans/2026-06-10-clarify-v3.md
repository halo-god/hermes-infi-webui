# Clarify v3: 基于 AskUserQuestion 模式的统一人工介入系统

> **Goal:** 将 Hermes 的 clarify 系统从双通道(Redis LIST + session/update)统一为 runner 中央调度的 tool_use 模式，借鉴 Claude Code 的 AskUserQuestion 设计。

**Architecture:** Runner 作为唯一调度器，拦截 agent 的 clarify tool_call，通过 SSE 推送结构化问题到前端，用户通过 ConfirmModal 回答，结果注入回 agent session。废弃 Redis LIST 侧通道。

**Tech Stack:** Python asyncio, Redis (SET+PUBLISH only, 不再用 LIST/BLPOP), Vue 3, SSE

---

## 现状分析

### 当前双通道架构

```
通道1 (legacy): Agent → clarify_callback → Redis LIST RPUSH → Runner LPOP → SSE → 用户 → Redis SET → Agent BLPOP
通道2 (native): Agent → session/update(confirmation_request) → Runner on_update → SSE → 用户 → Redis SET → Runner → Redis SET → Agent
```

**问题:**
1. 两条路径代码重复，逻辑分散在 `runner.py` 的 3 个方法 + `redis.py` 的 4 个函数
2. `clarify_callback` 阻塞 agent 线程，runner 必须轮询 Redis LIST（每秒 LPOP）
3. Redis 用了 5 个 key 做同一件事（`hermes:clarify:req:*`, `hermes:clarify:resp:*`, `hermes:clarify_pending:*`, `confirm:*`, `confirm_notify:*`）
4. `clarify_protocol = "dual"` 配置增加复杂度

### Claude Code 的做法

```
Agent → tool_use(AskUserQuestion) → canUseTool 回调 → 前端渲染选项 → 用户选择 → tool_result → Agent 继续
```

**核心区别:** 没有侧通道，没有轮询，没有阻塞。Runner 拦截 tool_call 事件，同步等待用户响应后注入结果。

---

## 目标架构

```
Agent 调用 clarify 工具
  ↓ (ACP session/update 或 tool_call 事件)
Runner 拦截
  ├─ 发送 SSE: clarification_request (带结构化选项)
  ├─ Redis SET: clarify:{conv_id}:{request_id} → pending
  └─ 后台任务: 等待 Redis PUBLISH clarify_response:{conv_id}
       ↓
前端 ConfirmModal 渲染
  ├─ 选项按钮 (label + description)
  ├─ 自由输入框
  └─ 跳过按钮
       ↓
用户选择 → POST /conversations/{id}/clarify
  ├─ Redis SET: clarify:{conv_id}:{request_id} → {choice}
  └─ Redis PUBLISH: clarify_response:{conv_id} → request_id
       ↓
Runner 后台任务收到通知
  ├─ SSE: clarification_response (前端关闭弹窗)
  └─ 注入 tool_result 到 agent session
       ↓
Agent 继续执行
```

## 改造计划

### Phase 1: 后端统一 (runner.py + redis.py)

#### Task 1.1: 简化 Redis key 设计

**目标:** 从 5 类 key 减少到 2 类

| 新 Key | 用途 | TTL |
|--------|------|-----|
| `clarify:{conv_id}:{request_id}` | 存储请求+响应状态 | 600s |
| `clarify_notify:{conv_id}` | Pub/Sub 通知 | - |

**废弃:**
- `hermes:clarify:req:{session_id}` (LIST, 不再需要)
- `hermes:clarify:resp:{session_id}:{id}` (LIST, 不再需要)
- `hermes:clarify_pending:{session_id}` (SET, 合并到新 key)
- `confirm:{conv_id}:{request_id}` (重命名)
- `confirm_notify:{conv_id}` (重命名)

**文件:** `backend/app/core/redis.py`
- 新增 `clarify_key(conv_id, request_id) -> str`
- 新增 `clarify_notify_key(conv_id) -> str`
- 重命名 `wait_for_confirmation` → `wait_for_clarify_response`
- 重命名 `respond_to_confirmation` → `respond_to_clarify`
- 保留旧函数名作为 alias（向后兼容）

#### Task 1.2: Runner 统一拦截逻辑

**目标:** 废弃 `_pop_clarify_request` 轮询，统一用 `on_update` 拦截

**文件:** `backend/agent_runner/runner.py`

改造 `on_update` 回调:
- 拦截 `kind == "tool_call"` 且 `tool_name == "clarify"` (或 `kind == "confirmation_request"`)
- 提取结构化数据: `{question, options: [{label, description}], allow_free_text, risk_level}`
- 发送 SSE `clarification_request`
- 启动 `_wait_and_clarify()` 后台任务

废弃:
- `_pop_clarify_request()` 方法（不再轮询）
- `_handle_clarify_request()` 方法（合并到 on_update）
- prompt loop 中的 `while not prompt_task.done()` 轮询逻辑

保留:
- `_wait_and_unblock_clarify()` → 重命名为 `_wait_and_clarify()`
- `_deliver_clarify_response()` → 简化
- `_record_clarify()` / `_update_clarify()` (审计日志)

#### Task 1.3: Agent 侧适配

**目标:** 确保 hermes-agent 的 clarify 工具走 session/update 通道

**文件:** `hermes-agent` 侧 (外部依赖，本次只记录改动点)
- `acp_adapter/session.py`: `_acp_clarify_callback` 改为发送 `session/update` 而非 Redis LIST RPUSH
- 废弃 `hermes:clarify:req:{session_id}` 的写入

**本次改动范围:** 只改 runner 侧，agent 侧保持兼容（dual 模式下两种都消费）

### Phase 2: API 层统一

#### Task 2.1: 重命名 confirm endpoint

```
POST /{conversation_id}/confirm  →  POST /{conversation_id}/clarify
```

保持旧 endpoint 作为 alias（向后兼容）。新 endpoint 接收:

```python
class ClarifyRequest(BaseModel):
    request_id: str
    choice: str
```

#### Task 2.2: 统一审批和 clarify 的 API 模型

**文件:** `backend/app/api/v1/conversations.py`, `backend/app/services/conversation_service.py`

审批（工具权限）和 clarify（人工介入）都通过同一个 `POST /clarify` 提交，`request_type` 字段区分:
- `request_type: "clarify"` — agent 问问题
- `request_type: "approval"` — agent 请求工具执行权限 (future)
